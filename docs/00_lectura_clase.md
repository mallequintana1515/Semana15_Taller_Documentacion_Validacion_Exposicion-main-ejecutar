# Lectura de clase - Semana 15

## Documentacion tecnica, bitacora de validacion y preparacion de exposicion

En esta semana vas a cerrar el proceso de desarrollo de un proyecto web desde una mirada mas profesional. Cuando un proyecto se entrega, no basta con mostrar que abre en el navegador. Tambien debe estar explicado, probado, corregido y preparado para que otras personas puedan entenderlo.

## 1. Documentacion tecnica

La documentacion tecnica es el conjunto de instrucciones y explicaciones que acompanan un proyecto. Sirve para que una persona pueda saber que hace el proyecto, como se instala, como se ejecuta, que tecnologias usa y como esta organizada su estructura.

En proyectos de software, un archivo muy usado para documentar es `README.md`. En este taller vas a completar `README_ENTREGA.md`, que sera tu documentacion tecnica final.

Una buena documentacion debe responder:

- Que hace el proyecto.
- Para que fue creado.
- Que tecnologias utiliza.
- Como se instala.
- Como se ejecuta.
- Que carpetas y archivos contiene.
- Que funcionalidades ofrece.
- Que pruebas se realizaron.
- Que errores fueron corregidos.
- Donde estan las evidencias.

## 2. Bitacora de validacion y correccion

La bitacora es el registro organizado de tu proceso de pruebas. No se trata solo de decir "funciona". Debes explicar que probaste, que esperabas que ocurriera, que paso realmente, que error encontraste y como lo corregiste.

Ejemplo:

| Prueba | Resultado esperado | Resultado obtenido | Correccion |
|---|---|---|---|
| Enviar formulario vacio | Mostrar errores | Se guardaba informacion incompleta | Se agrego validacion de campos obligatorios |

Esta bitacora demuestra tu proceso tecnico.

## 3. Evidencias

Las evidencias son capturas o registros que prueban que hiciste las verificaciones. Puedes tomar capturas del navegador, la consola, la base de datos local, el formulario validando o la vista responsive.

Una buena evidencia debe tener relacion clara con una prueba.

## 4. Preparacion de exposicion

Una exposicion tecnica debe ser breve, clara y demostrativa. No consiste en leer todo el codigo. Consiste en explicar el proyecto, mostrar su funcionamiento, justificar tus decisiones y presentar las pruebas realizadas.

Una buena exposicion incluye:

1. Nombre y objetivo del proyecto.
2. Problema o necesidad que atiende.
3. Tecnologias utilizadas.
4. Demostracion del funcionamiento.
5. Validaciones implementadas.
6. Errores encontrados y correcciones.
7. Evidencias.
8. Aprendizaje final.

## 5. Idea central de la semana

Un proyecto de software se fortalece cuando funciona, pero tambien cuando esta documentado, probado, corregido y puede ser explicado por quien lo desarrollo.
